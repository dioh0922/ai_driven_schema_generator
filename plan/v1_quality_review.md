# V1 品質・現実性レビュー報告書 (@Realistic)

作成日: 2026-06-05  
レビュー者: Gemini CLI (@Realistic)

`@Inverter` による破壊的視点での指摘を受け、本報告書では「継続的な開発と運用」に耐えうる品質であるかを、シニアエンジニアの視点でレビューした。

---

### 🔎 コードの懸念点

#### 1. エラーハンドリングの具体性不足
- **現状**: `generateSchema` 内で発生したエラーを `src/index.ts` でキャッチし、一律 500 エラーとして返している。
- **懸念**: AI のクォータ制限（429）や認証エラー（401/403）など、ユーザー側で対処可能なエラーとサーバー側の致命的なエラーが区別されておらず、トラブルシューティングが困難。

#### 2. 静配信の実装
- **現状**: `fs.readFile` を使った独自の実装（`servePublic`）を行っている。
- **懸念**: キャッシュヘッダー（ETag, Cache-Control）の欠如、Range リクエストへの非対応など、パフォーマンスと標準準拠の面で劣る。`@fastify/static` などの定評あるライブラリを使用すべき。

#### 3. 型定義の配置と再利用性
- **現状**: `src/types/schema.ts` に定義されている。
- **懸念**: サーバーと（将来的な）フロントエンドで共有されるべき型が混在している。また、`satisfies ErrorResponse` などは良いが、API のリクエスト/レスポンス全体のスキーマ（JSON Schema）が定義されていないため、Fastify の強力なバリデーション機能を活かせていない。

#### 4. プロンプト管理の疎結合化
- **現状**: `src/core/generator.ts` にテンプレート文字列としてハードコードされている。
- **懸念**: プロンプトは「コード」ではなく「設定/データ」に近い。将来的に複数のテンプレートを使い分けたり、調整したりする際に、ロジックと密結合していると変更のインパクトが大きくなる。

---

### ### [リファクタリング案]

1.  **Fastify エコシステムの活用**:
    - `@fastify/static` を導入し、`public/` ディレクトリを安全かつ高速に配信する。
2.  **型安全なスキーマバリデーション**:
    - Fastify の `schema` オプションに JSON Schema を定義し、API 入力を自動でバリデーションする（これにより `@Inverter` 指摘の文字数制限も解決可能）。
3.  **カスタムエラークラスの導入**:
    - `AppError` クラスを作成し、HTTP ステータスコードとエラーメッセージを紐付ける。
4.  **出力管理の改善**:
    - 出力先を `output/[request_id]/` のように動的に生成し、並列実行時の競合を回避する。

---

### 💰 現実的な判断
V1 リリースとしては、**「最小限の修正で致命的リスクを回避する」** ことを優先すべきである。完全なクリーンアーキテクチャへの移行は V2 以降とし、まずは `@Inverter` 指摘の防御策を最優先で適用することを推奨する。
