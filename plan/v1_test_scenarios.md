# V1 堅牢性テスト観点リスト (@Inverter)

修正後のシステムが「本当に壊れないか」を確認するための、意地悪なテスト観点一覧。

## 1. 入力バリデーション (Validation)
- [ ] **境界値テスト**: 2000文字丁度が成功し、2001文字が 400 エラーで即座に拒否されるか。
- [ ] **空入力テスト**: `""` や `" "` (スペースのみ) が拒否されるか。
- [ ] **型汚染テスト**: `{ "prompt": 123 }` (数値) や `{ "prompt": ["text"] }` (配列) を送った際に Fastify が型エラーで弾くか。

## 2. セキュリティ (Security)
- [ ] **トラバーサル試行**: `/..%2f..%2f.env` のようなエンコードされたパスで、プロジェクトルートの秘匿ファイルが露出しないか。
- [ ] **存在しないファイル**: `/non-existent.js` が 404 を返し、サーバーのスタックトレースや内部パスを露出させていないか。

## 3. 並列・リソース管理 (Concurrency & Resource)
- [ ] **ディレクトリ分離**: 同時に複数のリクエストを投げたとき、`output/` 内にそれぞれ別のディレクトリが生成されているか。
- [ ] **ファイル内容の独立**: 並列リクエストの結果、`schema.sql` の内容が他のリクエストのものと混ざっていないか。
- [ ] **ディスク圧迫の観測**: 生成を繰り返した際にディスクがどれくらい消費されるか（運用クリーンアップの必要性の確認）。

## 4. AI 応答の不確実性 (AI Non-determinism)
- [ ] **区切り文字欠落**: AI が `---` を出力しなかった場合、システムがパニック（クラッシュ）せず、適切な 500 エラーメッセージを返せるか。
- [ ] **余計なMarkdown**: AI が ```sql ... ``` のように Markdown 装飾を付けて返してきた場合、パースが壊れないか（現状の `split` ロジックの限界確認）。
